# 진행순서 (PR 단위)

이 문서는 개인 프로젝트 `ai-rag-platform`을 “실무처럼 PR 단위로 쪼개서” 진행하기 위한 체크리스트입니다.

## AI 모델/프로바이더 결정
- 현재 구현 기준은 `OpenAI API`로 확정
- MVP 모델
  - 임베딩: `text-embedding-3-small`
  - 답변 생성: `gpt-4o-mini`
- 확장 방향: 추후 `Azure OpenAI`로 전환 가능하도록 Provider 분리 유지

## PR-00 프로젝트 부트스트랩 (로컬 실행 기준점)
- 목표
  - 로컬에서 **DB + 백엔드가 뜨는 기준점** 만들기
- 포함
  - 폴더 구조: `ai-rag-platform/{backend,frontend,infra}`
  - 로컬 DB: `postgres + pgvector` (Docker Compose)
  - 백엔드 DB 연결 설정
  - 헬스 체크 확인
- 완료 기준
  - `docker compose up -d` 후 DB `healthy`
  - `.\gradlew bootRun` 성공
  - `Invoke-RestMethod http://localhost:8080/actuator/health` → `UP`

## PR-01 프론트 부트스트랩 (Vue 3 + Vite)
- 목표

- 포함
  - Vue 3 + Vite 프로젝트 생성
  - `Vue Router`, `Pinia`, `axios` 연결
  - 환경변수: `VITE_API_BASE_URL`
  - “헬스체크 페이지” 1개 (버튼 클릭 → `UP` 표시)
- 완료 기준
  - `npm run dev` 성공
  - 프론트에서 백엔드 health 호출 성공

## PR-01.1 프론트 UI 베이스 (Naive UI + 레이아웃 골격)
- 목표
  - 이후 PR(로그인/문서/질의 화면)이 “빨리, 일관되게” 만들어지도록 UI 기반을 먼저 깔기
- 포함
  - UI 라이브러리 도입: `naive-ui`
  - App 레벨 Provider 적용: `NConfigProvider`, `NMessageProvider`(전역 토스트/알림 기반)
  - 기본 레이아웃 골격
    - 상단바(프로젝트 타이틀)
    - 콘텐츠 컨테이너(최대 폭/여백 규칙)
  - 공통 UX(최소)
    - 로딩/에러 표시 패턴 정하기(버튼 로딩, 에러 알럿 등)
  - 헬스체크 페이지 UI 적용
    - `NCard`, `NButton`, `NTag`, `NAlert` 등으로 교체
- 완료 기준
  - 헬스체크 화면이 Naive UI 컴포넌트로 렌더링됨
  - 버튼 클릭 → `UP` 표시 + 실패 시 에러 알럿 표시
  - (개발 편의) 로컬에서는 프록시(`/api`) 또는 CORS 설정 중 하나로 호출 가능

## PR-07 단발 질의(RAG) + 선택 문서 멀티
- 목표
  - 핵심 기능: **선택한 문서들만** 근거로 단발 답변 생성
- 포함
  - 질의 API: `question` + `documentIds[]` (멀티 선택)
  - 권한 검증: 선택 문서가 전부 내 소유인지 체크
  - 벡터 검색: 선택 문서 청크만 대상으로 Top-K
  - 답변 생성 + sources(근거 문서/청크 일부/스코어) 반환
- 완료 기준
  - 선택 문서 묶음을 바꾸면 답변 근거가 달라짐
  - sources가 함께 내려옴

## PR-08 프론트 문서 멀티 선택 + 질의 화면
- 목표
  - 사용자 플로우 완성: “문서 선택 → 질문 → 답변(+근거)”
- 포함
  - 문서 리스트 체크박스(검색/필터 포함)

  - 답변 출력 + sources 표시
- 완료 기준
  - end-to-end로 단발 RAG 동작
